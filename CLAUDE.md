# Lemma — CLAUDE.md

## Проект

PWA-приложение для изучения английских слов (бывш. WordPunk).
React + TypeScript + Vite + IndexedDB (idb). Без бэкенда — всё локально.

**Цель проекта: веб-приложение, доступное по ссылке (GitHub Pages). БЕЗ публикации в App Store / Google Play.** Нативные сторы и Capacitor-обёртка — архивированы (см. Stage E).

GitHub: https://github.com/emorozoff/lemma
Deploy: https://emorozoff.github.io/lemma/

## Стиль (с v0.9 — минимализм, без киберпанка)

- Шрифт: IBM Plex Mono везде (`--font-ui`/`--font-display`/`--font-logo`), из Google Fonts (index.html). (`--font-logo` оставлен отдельной переменной на случай, если вордмарку захочется отдельный шрифт.)
- Палитра: чёрный/белый + **один** акцент `--accent: #e5342b` (красный). Токены в `:root`, тинты через `color-mix(in srgb, var(--accent) N%, transparent)` (`--accent-soft`, `--accent-line` и т.д.).
- Семантика цвета: **красный = активно/верно/бренд**, **серый (`--text-muted`) = неактивно/неверно**. Легаси-алиасы: `--accent-green`/`--accent-amber` → `--accent`; `--accent-red` → `--text-muted`.
- Светлая/тёмная тема: атрибут `[data-theme]` на `<html>`. **Главная тема — СВЕТЛАЯ (по умолчанию).** Тёмная (`[data-theme="dark"]`) включается ТОЛЬКО вручную в Настройках. Режим «авто/system» убран (v1.11); legacy-значение `lemma_theme='system'` трактуется как светлая. `ThemeProvider` + `lib/theme.ts` (ThemeMode = 'light'|'dark', `THEME_ORDER=['light','dark']`), переключатель в Настройках, ключ localStorage `lemma_theme`, pre-paint скрипт в `index.html` (без мигания; дефолт light).
- Тёмная: фон `#0a0a0a`; светлая: фон `#fff`. Без неона и глитч-хроматики. Едва заметная горизонтальная скан-текстура — в фоне `body` (слоями: gradient над `--bg`), цвет `--scanline-color`, темо-зависимая, **под всем контентом** (кнопки/карточки на surface-фоне перекрывают её, текст выше).
- Вордмарк: `lemma_` (строчные + подчёркивание). Кнопка флага — только красная иконка `⚑` (opacity 0.45 → 1 при отметке).

### Режим быстрого ввода — реализовано в v0.96 (тумблер в Настройках)
Переиспользует механизм финала: `isFinale` включает `isFastInputEnabled()`, ответ через `handleManualSubmit` → правильно = `processFinaleAnswer` (archived), ошибка = мягкий сброс на завтра. В режиме берётся 60 новых карточек за сессию вместо 20. Когда ВКЛючён:
- Каждая карточка показывает русское слово; пользователь сам **вписывает перевод на английском** (переиспользует UI ручного ввода с буквенными ячейками, как на финале lvl 4, но для ВСЕХ карточек).
- Правильно вписал → слово **сразу засчитывается как выученное** (минуя пошаговый SRS): +1 в «знаю слов» немедленно.
- Зачем: на старте все слова слишком простые — приходится каждый раз зажимать вариант и вводить слово, хотя ошибиться невозможно. Режим даёт быстро «пролистать» тривиальные слова.
- Пользователь **сам выключает** режим, когда дойдёт до сложных слов (возвращается обычный SRS с выбором из 4).

- **A — Минимум (3 шага):** ответ (тап перевода) → Настройки (выбери темы, без них слов нет) → «знаю слов».

Ниже — исходные правила/решения по каждой задаче (в силе).

**1. Дедуп лемм (одна карточка на базовую форму).** Изучаем только леммы (kid, не kids; get, не getting; low, не lower).
- Удаляем РЕГУЛЯРНЫЕ словоформы, если базовая лемма уже есть: -s/-es (мн.ч./3 л.), -ed, -ing, -er/-est.
- -ing-формы (герундий/причастие) убираем ВСЕГДА, даже у неправильных глаголов (telling→tell, seeing→see) — это предсказуемая форма, не лемма. ⚠️ ИСКЛЮЧЕНИЕ: настоящие -ing-СЛОВА, которые сами по себе леммы-существительные (morning, thing, king, ring, evening, ceiling, building/meeting/feeling в значении существительного) — ОСТАВЛЯЕМ. Детект: стрип -ing (+ удвоение согласной / возврат -e: getting→get, making→make) → если база-глагол есть карточкой → кандидат на удаление; ложные базы (morn, th) и самостоятельные ноун-значения → LLM/ручная проверка.
- Тест-кейсы из реального теста (регрессия при реализации): kid/kids, picture/pictures (регулярное мн.ч. → оставить лемму picture), telling→tell. Caveat: пользователь мог заархивировать форму (pictures) — при удалении формы его прогресс по ней «повиснет», но прогресс по лемме сохраняется.
- НЕ трогаем разные ЗНАЧЕНИЯ одного написания (row=ряд/грести, bank=банк/берег) — это не дубли.
- Синонимы-разнокоренные (mom/mum/mommy/mother) — это НЕ словоформы; курируем отдельно (оставить 1–2 стандартных, лишние убрать) + см. п.3.
- РЕШЕНО (пользователь): неправильные формы глагола держим ОТДЕЛЬНЫМИ карточками, ЕСЛИ написание отличается (see/saw/seen, go/went). Если форма пишется как база (cut/cut, put/put, read/read) — дубль, лишнюю убираем. Тот же принцип на прочие нерегулярные формы с другим написанием (children, men, feet, better/best) — оставляем; предсказуемые регулярные (kids, gets, getting, lower, walked) — схлопываем в лемму. ([[project_word_forms]] в силе, уточнён.)
- Как: (а) программный скан кандидатов (стрип окончаний → есть ли база) — дёшево, список на ревью; (б) LLM-проверка каждого на ложные срабатывания (glasses≠glass, news, series) и «другое значение»; (в) удаляем подтверждённые строки.
- Профилактика на будущее: vitest-тест/скрипт, падающий при появлении пары «лемма + её регулярная форма», чтобы дубли не заводились снова.

**2. Сокращение базы — по ПОЛЕЗНОСТИ, не по уровням (РЕШЕНО пользователем).** НЕ режем механически по freqLevel/difficulty. LLM сама оценивает каждую карточку: пригодится ли слово обычному человеку / знает ли его рядовой носитель / нужно ли учащемуся. Режем нишевый жаргон, узкотехническое, сверхредкое, архаизмы (пример пользователя: деталь машины, название которой обычный носитель не знает — убрать). freqLevel/difficulty — лишь ВХОДНОЙ СИГНАЛ для модели, не критерий. Целевого размера нет — сколько останется полезных, столько и останется.
- Как: воркфлоу-проход по каждой карточке (чанки) → {keep | cut} + причина + категория (жаргон / узкотехническое / устаревшее / сверхредкое / имя собственное) + confidence; на cut — adversarial-проверка (не срезать полезное); защищённый список очень частотных — никогда не резать. Архитектура как в аудите переводов. В конце — список «что удалили» на ревью.

**3. Похожие/синонимичные дистракторы.** ДВА СЛОЯ (баги со скринов относятся к разным):
- Слой 1 — СЛОВОФОРМЫ (house→дом + «дома»; «мать»/«матери»; рука/руки). Должен ловить `isSameLemma`, но для en-ru он требует общий префикс ≥5 символов → короткие слова (дом=3 буквы, стем мать=3) проскакивают. Фикс — улучшить `isSameLemma` для русского: либо «короткое — полный префикс длинного» + порог по доле общего стема, либо подключить русский стеммер (Snowball/Porter) и сравнивать стемы. КОД-ОНЛИ, без правок базы, можно сделать ОТДЕЛЬНО и быстро. Снимает кейс «дом/дома».
- Слой 2 — СИНОНИМЫ-разнокоренные (мама vs мать, mom vs mother): разные леммы, стеммер не поможет. Нужны «группы значения» (senseKey: мама/мать/мамочка = одна группа) → generateOptions исключает дистракторы из группы верного ответа. Группы проставит LLM в ходе ревизии (или заполнить пустое поле `synonyms` и учитывать его). Снимает кейс «мама/мать».
- ТЕСТ-КЕЙСЫ (баги со скринов, для регрессии при реализации):
  - Слой 1 (словоформы): house→дом + «дома»; stand→стоять + «стоял»/«стоит» (стоять/стоит расходятся на 4-й букве я/и → «полный префикс» НЕ ловит → нужен именно русский стеммер, не префикс-эвристика); «мать»/«матери».
  - Слой 2 (синонимы): mom→мама + «мать»/«матери» (мать — другая лемма).
- СТАТУС: **Слой 1 — РЕАЛИЗОВАН в v1.27** (`isSameLemmaRu` в srs.ts: стем ≥3 + флективный хвост; склонна схлопывать = безопасно). Снял дом/дома, стоять/стоял/стоит, рука/руки. Тесты в srs.test.ts. **Слой 2 (синонимы мама/мать) — НЕ сделан**, идёт с ревизией базы (senseKey-группы) ИЛИ быстрый костыль (ручной список кластеров) по запросу.

Связь: п.2 и п.3 удобно одним LLM-проходом (по карточке: keep/cut + senseKey + дубль-флаг). Механика как в ревизии переводов: правки прямо в words.ts, стабильные ID, в конце — список «удалили/сгруппировали».

### (история плана) Алгоритм подбора слов — убрать «ожидание по времени»

## UX-принципы

- **Зона большого пальца**: все интерактивные элементы (кнопки ответов, кнопка ДАЛЕЕ) должны быть внизу экрана, в зоне досягаемости большого пальца на телефоне. Карточка со словом — сверху/в центре, кнопки — снизу.

- **Sentence-first обучение (навсегда)**: все карточки показывают предложение как главный элемент, изучаемое слово выделено внутри предложения. Пользователь учит слова в контексте, не в вакууме. Для многозначных слов (lose = потерять/проиграть) делаем отдельные карточки с разными примерами — каждая раскрывает одно конкретное значение. Единый UX для всех карточек, без исключений.

## База слов

- Целевого размера больше нет (убран после ревизии v2, см. выше) — сколько останется полезных карточек после ревизии/Фазы 3, столько и останется. Текущий размер: **5072 карточки**.
- Одно английское слово может быть в нескольких карточках, если имеет разные значения (каждое значение — отдельная карточка с примером, раскрывающим именно это значение).
- Счётчик уникальных слов в шапке считается по английскому слову, а не по количеству карточек.

### Удаление карточек — ВАЖНО

При ревизии и удалении карточек **редактировать `src/data/words.ts` НАПРЯМУЮ**, а НЕ удалять строки из `data/raw/*.csv` с пересборкой.

Причина: `scripts/build-words.mjs` генерирует ID последовательно (`airport_001`, `airport_002`...). Если удалить строку из CSV и пересобрать, все последующие ID сдвинутся → у пользователей в IndexedDB прогресс «съедет» на соседние карточки (например, прогресс по `airport_044 layover` начнёт ссылаться на `airport_044 transit`). Прямое редактирование `words.ts` сохраняет стабильные ID.

Побочный эффект: `data/raw/*.csv` расходятся с `words.ts`. Это терпимо — `words.ts` де-факто становится source of truth. Если в будущем понадобится снова собирать из CSV, сначала надо рефакторить `build-words.mjs` на стабильные ID (например, хеш от `topicId+english+russian`).

## SRS v2 (с v0.83)

- **Уровни:** 0–4. Интервалы: `[0, 1, 3, 7, 14]` дней.
- **Уровень 0:** знакомство, 2 правильных подряд → lvl 1. Ошибка сбрасывает серию.
- **Уровни 1–4:** выбор из 4. Правильно → +1 уровень (для `difficulty` 1–2: +2). Ошибка → −1 (минимум 1).
- **Уровень 4 = финал.** ВСЕГДА ручной ввод (настройка убрана в v1.2). Строгость — по общей настройке «нестрогий ввод» (по умолчанию строго; точный ввод финализирует слово → архив).
- **Счётчик «знаю слов»:** level × 0.2 (1→0.2, 2→0.4, 3→0.6, 4→0.8), архив = 1.0, сумма округляется.
- Единый источник — `getKnownCount()` в `src/db/index.ts`; оптимистичные +1 в UI удалены.
